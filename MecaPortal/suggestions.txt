这个方案的核心思路是合理的：对于单份、结构化程度高（命令名精确、术语固定）的PDF手册，用 grep + read 代替向量检索/RAG，确实是性价比最高的工程选择——省去了embedding、向量库、chunk策略等复杂度，且检索精确度对"MoveLin"这类精确指令名反而更高。架构分层（提取→搜索→综合）逻辑清晰，落地步骤也具体可执行。

但在真正实施前，有几个关键技术细节和假设风险需要先验证/修正，否则方案在实际运行时可能"看起来对，跑起来不对"。

⚠️ 需要修正/验证的关键问题
1. Read 工具是否支持按行/按范围读取？（最关键，影响整体可行性）
方案里 "Grep定位 → Read上下文" 这一步，隐含前提是：Claude Code 的 Read 工具支持 offset + limit 参数（即根据Grep返回的行号，只读周围一段，而不是整个454KB文件）。
 - 必须在 CLAUDE.md 里写明具体操作模式，例如：
    1. Grep -n "MoveLin" manual.txt  → 得到行号 N
    2. Read manual.txt with offset=N-30, limit=80
    不能只写"阅读上下文"这种模糊指令，否则Claude可能每次都整读全文，失去分层检索的意义。
 - 建议先手动测一次：Grep 拿到行号后，Read 加 offset/limit 能否精准返回该段落。

2. pdftotext -layout 对表格/多列排版的还原质量未经验证
机械臂手册里大量是指令参数表（参数名/类型/单位/范围）。-layout 模式在多栏PDF或复杂表格上容易出现：
 - 列错位、跨行拼接错误
 - 表头和表体行被分割到不相邻的文本行

建议：提取后人工抽查5~10个典型指令表格（比如 MoveLin 参数表），确认没有错位。如果错位严重，考虑：
 - 对比不加 -layout 的效果
 - 或对"指令速查"类章节额外做一次人工整理成独立的 markdown 表格文件（比如 commands.md），作为高置信度补充源，Grep 优先命中这个小文件。

3. 触发关键词列表的维护成本被低估，且机制不够可靠
"列举15+类关键词"这种方式：
 - 容易漏（用户问法千变万化，比如"末端姿态怎么设"没有任何列表词但也该触发）
 - Claude 本身完全有能力判断"这是机械臂技术问题"，不需要精确关键词匹配

建议：把触发规则改成原则性描述而非枚举列表，例如：
"只要用户的问题涉及 Meca500 的具体指令、参数、坐标系、错误代码、编程细节，无论是否命中特定关键词，都应先搜索 manual.txt 再回答，不要凭记忆瞎答。"
关键词列表可以保留作为"示例"，但不要作为触发的唯一依据，否则维护负担大且不可靠。

4. 中英文术语不匹配问题（容易漏检索，方案未提及）
手册原文是英文，用户提问是中文（如"画圆"、"夹爪"、"奇异点"）。Grep是精确字符串匹配，中文词在英文手册里搜不到任何东西。
必须在CLAUDE.md搜索步骤里加一步：
    0. 先将用户的中文/口语化描述转换为可能对应的英文技术术语
   （画圆→circular/arc/interpolation, 夹爪→gripper, 奇异点→singularity）
    1. 再用转换后的英文关键词 Grep manual.txt
这是最容易被忽略但最影响命中率的一环。

5. 方案文档里预设了"标准答案"，可能误导验证标准
你在验证步骤里写道："预期：Meca500无原生画圆命令，需要多段MoveLin拟合"——这是你自己的先验判断，还没有真正检索过手册。如果手册里实际有 MoveCirc/圆弧插补类命令，这个预设就会误导测试判断（甚至误导实施者去"凑答案"）。

建议：验证阶段不要预设结论，只验证"Claude是否真的执行了搜索行为并引用了手册原文"，答案内容本身让检索结果说话。

6. 图片/图示信息会完全丢失
pdftotext 只提取文字，无法提取：
 - 坐标系示意图（轴向、正负号）
 - 机械结构尺寸图
 - 运动学图解

这类图对"运动控制"场景（比如TCP坐标系方向）往往是关键信息，纯文本会丢失。

建议：至少对"坐标系定义"、"TCP/WRF/FRF"相关的1~2张关键图，人工补充一段文字说明写进CLAUDE.md或单独的 coordinate_notes.md，作为兜底。

7. 权限配置语法需要实测，别假设格式正确
"Grep(* MecaPortal/search/manual.txt)" 这种带通配符+文件路径的权限写法，需要确认 Claude Code 的 settings.local.json 是否真的支持这种"参数级别"的allow规则（很多工具的permission只支持到"工具名级别"，不支持限定具体文件参数）。

建议：先用最简单的 "Grep", "Read", "Bash(pdftotext:*)" 这种粗粒度规则跑通，再考虑是否要精细化限制。

✅ 建议补充的优化项（非必须，但值得考虑）
1. 按章节拆分文件：370页/10900行单文件虽然Grep能搜，但如果按手册目录拆成 chapters/10_motion_commands.txt、chapters/06_coordinate_systems.txt 等，Grep结果自带"章节语境"，Claude综合答案时更聚焦，也方便日后升级为多PDF场景。这是"可选进阶"，如果验证下来单文件Grep效果够用，不必强求。
2. 指令速查表单独提取成结构化小文件：把所有指令名+简要说明+参数提取成一个 commands_index.md（几KB大小），作为Claude"第一优先"参照，Grep大文件作为"细节兜底"。这样能显著提升响应速度和准确率。
3. 速查表的行号问题：如果CLAUDE.md里的"章节速查表"标注的是行号，一旦 extract.sh 重新生成、PDF版本更新，行号会整体偏移，速查表就失效了。建议速查表只标印刷页码/章节号（人类可读性），不标行号；行号永远靠实时Grep获取，不要写死在CLAUDE.md里。

结论
方向对，工程选型合理，但不建议直接照单实施，请先按上面第1、2、4项（Read的offset/limit可行性、表格提取质量、中英术语转换）做小范围验证，这三项任何一个不成立，整套方案的检索准确率都会大打折扣。其余属于锦上添花或维护成本优化，可以在跑通MVP后再迭代。
